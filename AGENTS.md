# Операционный гайд для ИИ-агента (правки боевой системы)

Этот файл — инструкция для ИИ-ассистента (включая модели попроще, напр. Composer),
который вносит **ручные операционные правки** в живую систему расписания:
добавить/изменить сотрудника, отметить отсутствия, зафиксировать суточные и
догенерировать остальное и т.п.

> Архитектура приложения и нюансы фреймворка — в `web/AGENTS.md`. Здесь — только эксплуатация.

---

## 0. Золотые правила (читать всегда)

1. **Перед ЛЮБОЙ записью в БД — сделай снимок** (см. §3). Это мгновенный откат.
2. **Боевая БД — единственная.** Стейджинга нет. Ошибка = реальные последствия.
3. **Не коммить секреты.** Пароль root / S3-ключи в репозиторий не пишем. Они даются
   пользователем в сессии или лежат в `/opt/raspisoshlyapa/.env` на сервере.
4. **Имена сотрудников — это ключ.** В БД хранится фамилия (напр. `Кан`). Регистр и
   написание должны совпадать с тем, что уже есть. Сначала `list`, потом правки.
5. **После правки — проверь результат** (повтори `show`/`list`) и сообщи пользователю,
   что изменилось.
6. **Сложные/необратимые штуки** (удаление сотрудника, замена всей БД) — сначала
   проговори с пользователем, потом делай.

## 2. Как запускать операции (общий паттерн)

Скрипты-рецепты лежат в репозитории: `web/scripts/ops/*.cjs`. Они работают через
Prisma внутри контейнера. Том `./data` ↔ `/app/data` общий, поэтому скрипт кладём в
`data/` и запускаем внутри контейнера.

```bash
# на сервере: подтянуть свежие скрипты и положить их в смонтированный том
cd /opt/raspisoshlyapa && git pull --ff-only
cp web/scripts/ops/*.cjs data/

# запустить рецепт (единственный аргумент — JSON)
docker exec raspisoshlyapa-web-1 node /app/data/employee.cjs '{"action":"list"}'
```

Каждый рецепт принимает **один JSON-аргумент** — это удобно и однозначно для модели.
Действия `*show*`/`*list*` **только читают** — их можно вызывать без бэкапа.

## 3. Бэкап перед правкой (обязательно)

Локальный мгновенный снимок в общий том (откат за секунды):

```bash
docker exec raspisoshlyapa-web-1 sqlite3 /app/data/data.db \
  ".backup /app/data/_preedit_$(date +%F_%H%M%S).db"
```

Откат (короткий простой — корректно закрыть БД):

```bash
cd /opt/raspisoshlyapa
docker compose down
cp data/_preedit_<МЕТКА>.db data/data.db    # выбрать нужный снимок
docker compose up -d
```

Снимки `_preedit_*.db` периодически чисти, чтобы не копились:
`rm /opt/raspisoshlyapa/data/_preedit_*.db`

## 4. Модель данных (главное)

- **Employee** — сотрудник. Ключевые поля: `name`, `rate` (ставка, 0.5 = полставки),
  `targetRate` (целевая, держим в `[rate, maxRate]`), `maxRate` (потолок ≤ 2.0),
  `allowedPosts` (JSON-массив id постов), `modalities` (JSON, выводится из постов),
  `can24h` (допуск к суточным), `hospitalStartYear`/`careerStartYear` (для стажа,
  суточные — при общем стаже ≥ 5 лет), `seniority`.
- **Month** — месяц (`year`,`month`,`normHours`,`status`). Поле **`solverFixedSlots`**
  (JSON) — зафиксированные смены, которые солвер держит как ЖЁСТКОЕ ограничение.
- **Availability** — отсутствия сотрудника на месяц (`unavailableDays` JSON-массив дней).
- **Preference** — пожелания сотрудника на месяц (аппараты, смены, дни недели, минимум
  смен и т.д.). Обычно заполняют сами сотрудники через свой интерфейс.
- **ScheduleVersion** — версия расписания. `data` = `{ "<день>": { "<postId>": ["люди"] } }`,
  `status` = `draft` | `published`. Генерация создаёт новый draft.
- **ScheduleEdit** — журнал ручных правок ячеек.

### Форматы строк-меток

- На 12-часовом посту: просто `"Фамилия"` (напр. `"Кан"`).
- На **суточном** посту (24 ч): обязателен суффикс смены — `"Фамилия(с)"` (сутки),
  `"Фамилия(д)"` (день), `"Фамилия(н)"` (ночь).

## 5. Рецепты

Везде сначала делаем снимок из §3 (для операций записи).

### 5.1. Посмотреть сотрудников

```bash
docker exec raspisoshlyapa-web-1 node /app/data/employee.cjs '{"action":"list"}'
docker exec raspisoshlyapa-web-1 node /app/data/employee.cjs '{"action":"show","name":"Кан"}'
```

### 5.2. Добавить / изменить сотрудника

`posts` задаёт допуски и автоматически выводит модальности и `can24h`
(он станет `true`, если в наборе есть суточный КТ-пост `ssk1`/`kt_pb`).
Переданные поля меняются, остальные у существующего не трогаются.

> **Важно:** вход работника идёт по таблице `User` (login = фамилия в нижнем
> регистре, пароль `Боткин1!`). Скрипт `employee.cjs` при `upsert` сам создаёт
> учётку, если её нет — иначе фамилию «не найдёт» на входе. Если заводишь сотрудника
> напрямую через БД/Prisma, не забудь создать и `User`.

```bash
# новый КТ-доктор на одну ставку, без суточных
docker exec raspisoshlyapa-web-1 node /app/data/employee.cjs \
  '{"action":"upsert","name":"Кан","rate":1.0,"maxRate":1.5,"posts":["kt_ssk2","kt_2011","kt_2013","kt_4str"],"can24h":false}'

# отметить полставочником
docker exec raspisoshlyapa-web-1 node /app/data/employee.cjs '{"action":"upsert","name":"Корнилов","rate":0.5}'

# новый сотрудник со стажем с 2019 (суточные разрешены при наборе с ssk1/kt_pb)
docker exec raspisoshlyapa-web-1 node /app/data/employee.cjs \
  '{"action":"upsert","name":"Титов","rate":1.0,"hospitalStartYear":2019,"careerStartYear":2019,"seniority":7,"posts":["kt_pb","ssk1","kt_2011","kt_2013","kt_4str"]}'
```

### 5.3. Удалить сотрудника (необратимо — сначала спроси пользователя)

```bash
docker exec raspisoshlyapa-web-1 node /app/data/employee.cjs '{"action":"delete","name":"Ким"}'
```

### 5.4. Отметить отсутствия (отпуск/недоступные дни) на месяц

```bash
docker exec raspisoshlyapa-web-1 node /app/data/absence.cjs '{"action":"show","year":2026,"month":7,"employee":"Кан"}'
docker exec raspisoshlyapa-web-1 node /app/data/absence.cjs '{"action":"set","year":2026,"month":7,"employee":"Кан","days":[3,4,5,18]}'
docker exec raspisoshlyapa-web-1 node /app/data/absence.cjs '{"action":"add","year":2026,"month":7,"employee":"Кан","days":[20,21]}'
```

### 5.5. «Суточники подали смены → распредели остальных, не трогая их»

Это главный сценарий. Записываем суточные/готовые смены в `solverFixedSlots`,
затем запускаем генерацию — солвер сохранит зафиксированное и догенерит остальное.

```bash
# посмотреть текущие фиксы месяца
docker exec raspisoshlyapa-web-1 node /app/data/fixed-slots.cjs '{"action":"show","year":2026,"month":7}'

# добавить фиксы (merge — добавляет к существующим)
docker exec raspisoshlyapa-web-1 node /app/data/fixed-slots.cjs \
  '{"action":"merge","year":2026,"month":7,"slots":{"15":{"ssk1":["Иванов(с)","Гурова(д)"],"kt_pb":["Дохтова(н)"]},"16":{"kt_2011":["Кан"]}}}'

# заменить весь набор фиксов
# {"action":"set", ...}
# очистить
# {"action":"clear","year":2026,"month":7}
```

Затем **в админке** открой «Генерация» нужного месяца и запусти генерацию —
зафиксированные ячейки останутся как есть, остальное распределится. Если месяц
оказался «несовместимым» (infeasible), в UI есть мягкий режим (relax) — он выдаст
черновик с пометкой о пропусках.

> Запускать сам солвер из консоли не нужно: достаточно подготовить данные (фиксы,
> сотрудников, отсутствия), а кнопку «Сгенерировать» нажимает админ в интерфейсе.

### 5.6. Точечная правка готового расписания (не пересобирая месяц)

Если нужно вписать/убрать одного человека в уже сгенерированной версии — это удобнее
делать в редакторе админки (Расписание → редактировать): он сам пересчитывает часы и
ведёт журнал правок. Прямые правки `ScheduleVersion.data` через БД возможны, но тогда
не забудь пересчитать `employeeHours` — поэтому предпочтительнее UI.

## 6. Проверка после изменений

- Сотрудники: `employee.cjs '{"action":"list"}'` — число и ставки совпадают с ожиданием.
- Фиксы: `fixed-slots.cjs '{"action":"show",...}'`.
- Отсутствия: `absence.cjs '{"action":"show",...}'`.
- Расписание/часы — в админке после генерации.

Если что-то пошло не так — откат из снимка (§3).
